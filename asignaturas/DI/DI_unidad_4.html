1. Introducción a JasperReports

JasperReports es una biblioteca Java diseñada para generar informes de manera dinámica dentro de aplicaciones. Permite crear informes a partir de distintas fuentes de datos y exportarlos en múltiples formatos.

1.1. Componentes clave:

Archivos de informe

    • .jrxml → código fuente XML del informe. Editable en JasperSoft Studio o iReport.

    • .jasper → archivo compilado listo para ejecución; no se puede editar directamente.

Herramientas de diseño

    • JasperSoft Studio (actual y soportado) → permite diseño gráfico y edición de código fuente.

    • iReport (antigua) → versión clásica, todavía útil para aprender conceptos.

    • JasperViewer → clase Java que permite visualizar informes generados dentro de una aplicación.

Integración en proyectos

    • Se incluye la librería principal: jasperreports-x.x.x.jar.

    • Si se usan bases de datos, se requiere el driver JDBC correspondiente.

    • Carpeta típica: Reports/ para guardar .jrxml y .jasper.

1.2. Conceptos importantes:

  • JasperReports no es un IDE ni un servidor; es una biblioteca que genera informes.

  • Permite exportar informes a formatos como PDF, XLS, CSV, HTML, RTF, XML, PNG.

  • El flujo típico es: diseñar el informe → compilar a .jasper → generar el informe usando datos → exportar o mostrar.

2. Arquitectura de un informe y bandas (secciones)

  • Un informe en JasperReports está compuesto por bandas (secciones) que organizan la información visual y los datos extraídos de la fuente de datos.

  • Cada banda tiene propósito específico y puede repetirse o aparecer solo una vez según su tipo.

2.1. Bandas principales de un informe

Title

    • Aparece solo al inicio del informe, generalmente en la primera página.

    • Uso típico: título principal, logo, fecha, subtítulo.

Page Header

    • Aparece en la parte superior de cada página (excepto si el Title ocupa la primera página).

    • Uso típico: encabezados de columna, logotipos pequeños, información de página.

Column Header

    • Se coloca encima de cada grupo de columnas de datos.

    • Uso: etiquetas de columnas en tablas o grids de datos.

Detail

    • Se repite por cada registro de la fuente de datos.

    • Contiene los fields que muestran los datos extraídos.

    • Es la banda más importante para mostrar información de filas individuales.

Summary

    • Aparece una sola vez al final del informe.

    • Ideal para totales, gráficos o conclusiones.

Page Footer

    • Se muestra al final de cada página.

    • Uso típico: número de página, fecha de impresión, notas de pie de página.

2.2. Flujo de ejecución de bandas

    • JasperReports procesa las bandas en orden secuencial: Title → Page Header → Column Header → Detail → Summary → Page Footer.

    • Las bandas Detail y algunas variables se ejecutan por cada registro de datos, mientras que Title y Summary solo una vez.

2.3. Observaciones prácticas

    • Las bandas Summary y Title pueden contener gráficos, totales o cálculos agregados.

    • Las secciones de Header y Footer facilitan la uniformidad de cada página.

3. Fields, Variables y Parameters

    • JasperReports distingue entre datos directos de la fuente, valores calculados y valores de entrada externos.

    • Cada tipo cumple un propósito específico dentro del informe y su flujo de ejecución.

3.1. Fields (Campos)

    • Representan datos extraídos directamente de la fuente de datos (columna de base de datos o propiedad de un JavaBean).

Características

    • Se definen por nombre y tipo.

    • Se accede a ellos mediante getters en JavaBeans o claves en Map/JSON/XML.

    • Se muestran en la banda Detail para cada registro.

Ejemplo práctico

	<field name="nombre" class="java.lang.String"/>

3.2. Variables

    • Valores calculados a partir de Fields o de otras Variables.

    • Se utilizan para totales, promedios, conteos, máximos o mínimos.

Tipos de cálculo (Calculation Type)

    • SUM → suma valores numéricos.

    • COUNT → cuenta registros.

    • MAX / MIN → obtiene valores máximo o mínimo.

Características

    • Se actualizan según la banda de ejecución y el scope (grupo o informe completo).

    • Se muestran generalmente en Summary o en grupos.

Ejemplo práctico

	<variable name="totalPrecio" class="java.lang.Double" calculation="Sum">
		<variableExpression><![CDATA[$F{precio}]]></variableExpression>
	</variable>

3.3. Parameters (Parámetros)

    • Valores externos al informe que se proporcionan en tiempo de ejecución.

    • Se usan para filtrar datos, cambiar títulos, seleccionar rangos de fechas, etc.

Características

    • Se definen por nombre, tipo y opcionalmente un valor por defecto.

    • Pueden ser referenciados en queries, expressions de fields o variables.

Ejemplo práctico

	<parameter name="minPrecio" class="java.lang.Double"/>

3.4. Distinciones clave

    • Field: dato de la fuente de datos.

    • Variable: valor calculado dentro del informe.

    • Parameter: valor suministrado externamente al informe.

3.5. Observaciones prácticas

    • Variables y parámetros pueden intervenir en filtros y cálculos.

    • El uso correcto de Fields, Variables y Parameters es crucial para totales, gráficos y filtros dinámicos.

4. Tipos de fuentes de datos

4.1. Fuentes relacionales (SQL/JDBC)

    • Origen: bases de datos relacionales como MySQL, PostgreSQL, Oracle.

    • Consulta: SQL definida explícitamente en el <queryString> del informe.

    • Conexión: requiere driver JDBC, URL, usuario y contraseña.

    • Flujo: JasperReports envía la consulta a la base de datos, recibe filas y las itera en la banda Detail.

    • Ejemplo de query en .jrxml:

	<queryString language="SQL">
		<![CDATA[SELECT nombre, precio FROM productos WHERE precio > $P{minPrecio}]]>
	</queryString>

4.2. Fuentes no relacionales (en memoria)

    • Origen: datos generados dentro de la aplicación, como colecciones de JavaBeans, listas de Map, o estructuras JSON/XML.

    • Acceso: JasperReports itera la colección y obtiene datos mediante getters o keys.

    • Ejemplo con JavaBeans:

      JRBeanCollectionDataSource ds = new JRBeanCollectionDataSource(listaProductos);

    • Ejemplo con Map:

      JRMapCollectionDataSource ds = new JRMapCollectionDataSource(listaMapas);

4.3. Fuentes estructuradas (XML/JSON)

    • XML → se consulta mediante XPath.

    • JSON → se consulta mediante JSONPath.

    • JasperReports recorre la estructura y extrae los datos según las expresiones definidas.

4.4. Concepto clave

    • JasperReports no adivina el lenguaje de consulta; depende del tipo de datasource:

      • SQL → consultas explícitas.

      • JavaBeans/Map → iteración implícita con Fields.

      • XML/JSON → XPath o JSONPath.

5. Filtrado de datos

    • El filtrado permite restringir los registros que se muestran en un informe según condiciones específicas, mejorando la relevancia y el rendimiento.

5.1. Formas de aplicar filtros

Filter Expression

    • Expresión definida en JasperReports que devuelve true o false para cada registro.

    • Solo se incluyen los registros que cumplan la condición.

    • Ejemplo:

      $F{precio} > $P{minPrecio}

Parámetros

    • Los parámetros externos pueden modificar dinámicamente los filtros del informe (fechas, categorías, valores mínimos).

Consulta del dataset

    • Si la fuente es relacional, los filtros también se pueden aplicar en la cláusula WHERE de la consulta SQL.

5.2. Repercusiones del filtrado

    • Reduce la cantidad de datos procesados y visualizados.

    • Afecta totales, subtotales y variables calculadas.

    • Mejora la velocidad de generación del informe, especialmente en grandes volúmenes de datos.

5.3. Buenas prácticas

    • Combinar filtros en query, Filter Expression y parámetros según la necesidad.

    • Evitar filtrar grandes volúmenes de datos únicamente en JasperReports; es mejor hacerlo en la base de datos cuando sea posible.

6. Gráficos y visualización de datos

    • JasperReports permite incluir elementos gráficos en los informes para representar visualmente los datos.

    • Se pueden usar gráficos para comparar, resumir o mostrar tendencias de los datos.

6.1. Tipos de gráficos comunes

Barras (Bar Chart)

    • Compara valores numéricos entre categorías discretas.

    • Ejemplo: ventas por producto o localidad.

    Sectores / Pie Chart

    • Representa la proporción de cada parte respecto a un todo.

    • Ejemplo: porcentaje de ventas por categoría.

    Gantt

    • Muestra duración de actividades o tareas a lo largo de un tiempo total determinado.

    • Ejemplo: planificación de proyecto.

    Series temporales (Time Series Chart)

    • Representa datos medidos u observaciones ordenadas cronológicamente.

    • Ejemplo: evolución de ventas mes a mes.

6.2. Alimentación de datos en gráficos

    • Se define un Dataset interno o series que proporcionan:

      • Valores numéricos (Y-axis).

      • Categorías o fechas (X-axis).

    • Los campos y variables calculadas pueden formar parte del dataset.

6.3. Buenas prácticas

    • Elegir el tipo de gráfico según el objetivo: comparación (barras), proporción (pie), evolución temporal (series), planificación (Gantt).

    • Mantener claridad visual y consistencia con los datos mostrados en tablas.

7. Exportación de informes

    • La exportación permite guardar o compartir los informes generados en distintos formatos, facilitando su distribución y análisis.

7.1. Formatos soportados

    • PDF → estándar para impresión y presentación.

    • XLS / XLSX → hojas de cálculo para análisis.

    • CSV → datos tabulares para procesamiento externo.

    • HTML → visualización web.

    • RTF / XML / PNG → otros formatos según necesidad.

7.2. Consideraciones prácticas

    • La misma variable o campo puede aparecer en múltiples formatos sin cambios.

    • Algunos formatos requieren ajustes:

      • PDF mantiene estilos y diseño exacto.

      • XLS prioriza datos sobre formato visual.

      • PNG y gráficos requieren que los elementos estén correctamente dimensionados.

7.2. Buenas prácticas

    • Permitir generar informes y exportarlos en diferentes formatos desde la aplicación para máxima flexibilidad.

    • Evitar exportaciones excesivamente grandes sin filtrado previo para optimizar rendimiento.

8. Organización de proyectos y buenas prácticas

8.1. Estructura típica de un proyecto que usa JasperReports:

Reports/

    • Carpeta destinada a almacenar los informes: .jrxml (código fuente) y .jasper (compilados).

Clases/

    • Objetos Java que se usarán como datasource (JavaBeans).

Controladores/

    • Lógica de negocio que llama a los informes y gestiona parámetros y filtros.

Documentación/

    • Manuales, ejemplos de uso y notas técnicas.

8.2. Librerías necesarias

    • jasperreports-x.x.x.jar → biblioteca principal de JasperReports.

    • Drivers JDBC → necesarios para conexión a bases de datos relacionales.

    • Dependencias adicionales si se usan gráficos avanzados o exportación a formatos especiales.

8.3. Buenas prácticas de desarrollo

    • Mantener separados los informes, la lógica y los datos.

    • Usar nombres claros y consistentes para campos, variables y parámetros.

    • Editar .jrxml para modificaciones y compilar a .jasper antes de ejecutar.

    • Aplicar filtros y parámetros para mejorar eficiencia y personalización de informes.

    • Documentar los informes y su uso dentro del proyecto.

8.4. Consejo de integración

    • Diseñar informes modulares y reutilizables.

    • Incrustar gráficos y totales de manera que puedan adaptarse a diferentes formatos de exportación.
