Resumen compacto – Unidad 1 – Análisis de tecnologías para aplicaciones en dispositivos móviles

1. Limitaciones y características de los dispositivos móviles

1.1 Limitaciones físicas y energéticas

  • Batería: los dispositivos móviles funcionan con baterías de capacidad limitada. La duración depende del uso de CPU, GPU, pantalla y conectividad.

  • Throttling del procesador: los CPUs móviles reducen su frecuencia para ahorrar energía cuando la carga es alta.

  • Ventilación: generalmente no tienen ventilación activa, por lo que la disipación de calor depende de la carcasa y el diseño térmico.

  • Conectividad: no siempre están permanentemente conectados a la red eléctrica, lo que impone restricciones en la ejecución continua.

1.2 Limitaciones de memoria y almacenamiento

  • Memoria RAM limitada: no permite ejecutar aplicaciones pesadas simultáneamente sin afectar rendimiento.

  • Almacenamiento: limitado por la capacidad interna o microSD; no se dispone de almacenamiento “ilimitado” como en PCs.

  • Implica optimización del uso de recursos, especialmente para aplicaciones multimedia.

1.3 Limitaciones de interacción y usabilidad

  • Pantalla táctil: el diseño de la interfaz debe adaptarse a la manipulación con dedos, evitando elementos demasiado pequeños.

  • Tamaño reducido: limita la cantidad de información visible simultáneamente.

  • Entorno de uso: posibles reflejos en exteriores afectan la visibilidad.

  • Entrada limitada: normalmente no disponen de ratón físico ni teclado completo.

1.4 Limitaciones de conectividad y latencia

  • Transferencias de datos pueden experimentar latencias o interrupciones.

  • Consumo de datos y batería debe ser controlado en aplicaciones que dependen de la red.

1.5 Propósitos y ventajas

  • Los móviles permiten: • Movilidad: portabilidad y uso en cualquier lugar. • Comunicaciones: llamadas, mensajería, video. • Trabajo: productividad, gestión de información. • Entretenimiento: multimedia, juegos, redes sociales.

  • No son principalmente servidores de datos, aunque pueden cumplir tareas limitadas.

2. Sistemas operativos móviles

2.1 Principales sistemas operativos móviles

  • Android • Desarrollado inicialmente por Android Inc., adquirido por Google en 2005. • Sistema operativo basado en Linux Kernel, diseñado para dispositivos con pantalla táctil y recursos limitados. • Código abierto parcialmente (AOSP), permite personalización por fabricantes. • Orientado a smartphones, tablets, TV, wearables y otros dispositivos.

  • iOS • Desarrollado por Apple para iPhone, iPad y iPod Touch. • Sistema cerrado, con estricto control de hardware y distribución de apps a través de App Store. • Basado en Darwin (UNIX), con frameworks como Cocoa Touch para desarrollo de apps.

  • Windows Mobile / Windows Phone • Windows Mobile: SO anterior de Microsoft para PDAs y smartphones. • Windows Phone: sucesor (2010), con interfaz Metro, integración con servicios Microsoft y soporte limitado tras 2017.

  • BlackBerry OS • Desarrollado por BlackBerry Ltd. • Orientado a comunicaciones seguras y correo electrónico corporativo. • Actualmente obsoleto, reemplazado por BlackBerry 10 y sistemas Android en dispositivos nuevos.

  • WebOS • Inicialmente desarrollado por Palm, adquirido por HP. • Sistema multitarea basado en Linux, con interfaz por tarjetas. • Abandonado en 2011 para smartphones, pero usado posteriormente en Smart TV.

  • Tizen • Desarrollado por Samsung y Linux Foundation. • Basado en Linux, usado en wearables, TV y algunos smartphones.

  • Symbian • Desarrollado por Symbian Ltd., usado en Nokia y otros fabricantes. • Fue dominante antes de iOS y Android, actualmente obsoleto.

2.2 Características y enfoque de los SO móviles

  • Kernel: base del sistema, gestiona memoria, procesos, seguridad, drivers.

  • Framework de aplicaciones: proporciona APIs de alto nivel para el desarrollo (Activities, Services, Views).

  • Librerías nativas: C/C++ para multimedia, gráficos, bases de datos, etc.

  • Capa de aplicaciones: apps del sistema y de usuario.

2.3 Estado de los sistemas operativos

  • Activos: Android, iOS, Tizen (en wearables y TV).

  • Legado/Obsoletos: Windows Mobile, Windows Phone, WebOS, Symbian, BlackBerry OS.

  • Importancia académica: las preguntas de examen pueden referirse a SO obsoletos por historia, evolución o comparación de características.

2.4 Versiones de Android importantes

  • Cada versión de Android tiene un SDK y un API Level asociado.

  • Cambios destacados en versiones relevantes: • Android 6 (Marshmallow): modelo de permisos dinámico. • Android 8 (Oreo): restricciones en ejecución en segundo plano y ahorro de energía. • Android 10+: mejoras en privacidad, seguridad y compatibilidad con nuevos sensores y cámaras. • Versiones antiguas (Lollipop, KitKat, Jelly Bean): cambios en diseño de UI y soporte multiventana.

  • Compatibilidad app/emulador: el emulador debe tener API igual o superior a la minSdkVersion de la aplicación.

3. Enfoques de desarrollo para móviles

3.1 Desarrollo nativo

  • Definición: programar directamente usando las herramientas y APIs propias del sistema operativo.

  • Lenguajes y herramientas: • Android: Java o Kotlin + Android SDK + Android Studio. • iOS: Swift o Objective-C + Xcode + iOS SDK.

  • Ventajas: • Rendimiento óptimo y acceso completo a las APIs del dispositivo. • Mejor integración con hardware y características nativas.

  • Desventajas: • Código específico por plataforma → duplicación de esfuerzo si se quiere soportar múltiples SO.

3.2 Desarrollo multiplataforma

  • Definición: escribir una sola base de código que se compila o interpreta para varias plataformas.

  • Plataformas / frameworks: • Xamarin: C#, compila a nativo. • Flutter: Dart, compilación a código nativo ARM. • React Native: JavaScript, componentes nativos; parcialmente compilado a nativo.

  • Ventajas: • Reutilización de código, menor coste de desarrollo. • Distribución simultánea para Android e iOS.

  • Desventajas: • Limitaciones en acceso a algunas APIs nativas avanzadas. • Posible sobrecarga de rendimiento frente a desarrollo 100% nativo.

3.3 Multiplataforma basada en HTML5 / híbrida

  • Definición: apps construidas con HTML, CSS y JavaScript y empaquetadas en un contenedor nativo (WebView) usando frameworks como Cordova/PhoneGap, Ionic.

  • Ventajas: • Desarrollo rápido, reutilización de habilidades web. • Distribución multiplataforma con un solo código base.

  • Desventajas: • Rendimiento inferior frente a apps nativas. • Acceso limitado o indirecto a APIs nativas → requiere plugins.

  • Ejemplo de uso: apps simples de contenido dinámico, formularios, juegos ligeros.

3.4 Elección del enfoque

  • Depende de: • Objetivos del proyecto (performance vs rapidez de desarrollo). • Recursos disponibles (equipo especializado por SO o un equipo multidisciplinario). • Requerimientos de hardware/software específico del dispositivo.

4. Entornos de desarrollo (IDEs) y SDK

4.1 Características mínimas de un IDE para desarrollo móvil

Un IDE moderno debe incluir:

  • Editor de código avanzado • Resaltado de sintaxis, autocompletado, refactorización, navegación entre clases y funciones.

  • Compilador y empaquetador (build/deploy) • Genera el archivo ejecutable final (APK/AAB en Android), incluyendo firma y optimización.

  • Depurador • Permite ejecutar la app paso a paso, inspeccionar variables, detectar errores y analizar logs.

  • Emulador o capacidad de ejecución en dispositivos reales • Para probar apps sin necesidad de múltiples dispositivos físicos.

  • Editor de recursos • Gestión de layouts, strings, traducciones, imágenes básicas.

  • Integración con sistemas de control de versiones (VCS) • Git, GitHub, SVN, CVS para gestión de cambios y colaboración.

⚠️ Nota: herramientas como “analizador de archivos” no son indispensables ni específicas para desarrollo móvil.

4.2 Android Studio

  • IDE oficial para Android, basado en IntelliJ IDEA.

  • Funcionalidades clave: • Editor de código, depuración, emulación de dispositivos (AVD), Gradle build, integración con Git. • Acceso directo a SDK, herramientas de testing, emuladores externos (Genymotion) y dispositivos reales.

4.3 Gradle

  • Sistema de automatización de builds usado por Android Studio.

  • Archivos importantes: • build.gradle (Module: app): define dependencias, plugins, configuración de build específica del módulo. • build.gradle (Project): configuraciones generales del proyecto.

  • Funciones: • Gestionar dependencias externas. • Configurar tipos de build, versiones de SDK y firmas de apps.

4.4 Directorios principales de un proyecto Android

  • app/src/main/java: código fuente Java/Kotlin.

  • app/src/main/res: recursos compilables accesibles vía R (layouts, drawables, strings, styles).

  • app/src/main/assets: archivos sin procesar, leídos en ejecución (JSON, fuentes, datos estáticos).

  • app/build: directorio de compilación, generado automáticamente.

  • Otros archivos importantes: • settings.gradle: configuración del proyecto y módulos. • gradle-wrapper.properties: versión de Gradle utilizada.

4.5 SDK de Android

  • Conjunto de herramientas, librerías y utilidades para desarrollar aplicaciones.

  • Incluye: • Librerías nativas en C/C++ para multimedia, gráficos, base de datos. • Herramientas de línea de comandos: adb (Android Debug Bridge), avdmanager (gestión de emuladores).

  • Cada versión de Android tiene un SDK asociado identificado por su API Level.

4.6 Integración con control de versiones (Git/GitHub)

  • Clonación de proyectos: File > New > Project from Version Control.

  • .gitignore: indica archivos/directorios que no deben subirse al repositorio ( build/, .gradle/, local.properties ).

  • Permite colaboración y historial de cambios, fundamental en proyectos profesionales.

5. AndroidManifest y configuración de la app

5.1 Función del AndroidManifest.xml

  • Archivo central de configuración de la aplicación Android.

  • Contiene información esencial que el sistema necesita para ejecutar la app: • Nombre del paquete de la aplicación. • Componentes: Activities, Services, Broadcast Receivers, Content Providers. • Permisos que la app requiere ( <uses-permission> ). • Características del hardware/software requeridas ( <uses-feature> ). • Versiones mínimas y máximas de Android soportadas ( <uses-sdk> ).

⚠️ No se declara el tamaño final del APK, ni claves de firma en el manifiesto.

5.2 Declaración de permisos y features

  • Permisos (<uses-permission>) • Permiten que la app acceda a recursos sensibles (cámara, almacenamiento, contactos). • No garantizan la existencia del hardware; solo autorización para usarlo.

  • Features (<uses-feature>) • Indican requisitos hardware/software del dispositivo (cámara, GPS, Bluetooth). • Ejemplo:

    <uses-feature android:name="android.hardware.camera" android:required="true"/>

    • Permite filtrado de disponibilidad en Google Play y evita instalación en dispositivos sin la función.

5.3 Restricción de disponibilidad de la app

  • Métodos válidos: • Según hardware o software requerido ( <uses-feature> ). • Según configuración de pantalla, densidad, orientación. • Según versiones de plataforma ( minSdkVersion, maxSdkVersion ).

  • No se puede restringir por propósito del dispositivo (trabajo, entretenimiento, etc.).

5.4 Compatibilidad de versiones

  • minSdkVersion: API mínima requerida; impide instalación en dispositivos con versión inferior.

  • targetSdkVersion: indica la versión de Android para la que la app está optimizada.

  • Recomendación: el emulador o dispositivo debe tener API igual o superior a minSdkVersion para ejecutar correctamente la app.

6. Emuladores y ejecución en dispositivos

6.1 Android Virtual Device (AVD)

  • Definición: gestor oficial de emuladores de Android.

  • Permite simular dispositivos móviles con diferentes configuraciones de hardware y software.

  • Cada AVD incluye: • Perfil de hardware ( hardware-profile.xml ): pantalla, CPU, RAM, sensores, cámara, almacenamiento. • Imagen del sistema: versión específica de Android (API Level).

  • Se pueden crear tantos AVDs como se necesite, permitiendo probar diferentes resoluciones, versiones de Android y capacidades hardware.

6.2 Ejecución en emulador

  • Opciones de ejecución: • Emulador de Android Studio (oficial). • Emuladores externos como Genymotion.

  • Ventajas del emulador: • Pruebas rápidas sin necesidad de dispositivo físico. • Configuración de diversos escenarios (pantalla, sensores, red).

  • Recomendación: el emulador debe tener API igual o superior a la minSdkVersion de la app.

6.3 Ejecución en dispositivo real

  • Para ejecutar apps desde Android Studio en un dispositivo físico: • Activar Depuración USB en el dispositivo. • Conectar mediante cable USB. • Seleccionar el dispositivo en Android Studio como target.

  • Permite pruebas reales de rendimiento, sensores y conectividad.

6.4 Compatibilidad de versiones

  • Versión de app vs versión de emulador/dispositivo: • La app no puede ejecutarse en un sistema con API inferior a minSdkVersion. • targetSdkVersion indica la versión para la que la app está optimizada, no bloquea la instalación.

6.5 Resumen de buenas prácticas

  • Crear AVDs que reflejen los dispositivos más comunes de los usuarios.

  • Mantener múltiples perfiles para pruebas de compatibilidad.

  • Complementar pruebas en emulador con dispositivos reales, especialmente para multimedia, sensores y rendimiento.

7. Recursos de la app

7.1 Directorios principales de recursos

  • res/ (resources) • Contiene recursos estructurados y compilables accesibles vía R. • Subdirectorios típicos:

    • drawable/: imágenes y gráficos (PNG, SVG, XML).

    • layout/: archivos XML que definen la interfaz de usuario.

    • values/: cadenas ( strings.xml ), colores, estilos, dimensiones.

    • mipmap/: iconos de la aplicación. • Los recursos en res/ se compilan automáticamente y se accede mediante R.id o R.layout.

  • assets/ • Contiene archivos sin procesar que la app lee en ejecución. • Ejemplos: JSON, fuentes personalizadas, archivos de datos, música o videos no optimizados. • No se generan IDs en R; se accede mediante AssetManager.

7.2 Acceso a recursos desde código

  • Recursos compilados (res/): • findViewById(R.id.elemento) para elementos de UI. • getResources().getString(R.string.mi_cadena) para strings.

  • Recursos sin procesar (assets/): • AssetManager.open("archivo.json") para lectura de datos en ejecución. • Útil para archivos grandes o binarios, que no necesitan ser compilados.

7.3 Buenas prácticas

  • Separar recursos por tipo y densidad de pantalla (mdpi, hdpi, xhdpi).

  • Evitar duplicación de recursos; usar directivas de valores ( values/ ) para mantener consistencia.

  • Guardar en assets/ solo archivos que no requieren compilación ni acceso mediante R.

8. Distribución y publicación

8.1 Google Play

  • Definición: tienda oficial de aplicaciones para Android.

  • Funciones principales: • Distribución de aplicaciones a usuarios finales. • Gestión de actualizaciones automáticas. • Integración con pagos y suscripciones. • Filtrado de dispositivos según:

    • Hardware requerido ( <uses-feature> ).

    • Versión mínima de Android ( minSdkVersion ).

    • Región o país del usuario.

8.2 Control de disponibilidad de la aplicación

  • Restricciones que se pueden aplicar: • Hardware y software requerido: sensores, cámara, GPS, Bluetooth. • Configuración de pantalla: tamaño, densidad y orientación. • Versiones de plataforma: minSdkVersion y targetSdkVersion.

  • No es posible restringir según el propósito del dispositivo (trabajo, entretenimiento, comunicación).

8.3 Publicación

  • Pasos principales: • Generación de APK o AAB firmado con clave privada. • Pruebas de compatibilidad en emulador y dispositivos reales. • Subida a Google Play con información de la app, imágenes, descripción y categorización. • Configuración de restricciones de compatibilidad para filtrar dispositivos no soportados.

8.4 Buenas prácticas

  • Probar en múltiples versiones de Android y perfiles de dispositivo antes de publicar.

  • Mantener consistencia en permisos y features entre manifiesto y Google Play.

  • Usar control de versiones y herramientas de integración continua para gestionar releases.

9. Otros conceptos complementarios

9.1 Seguridad y firma de aplicaciones

  • Claves y keystore: • Cada app Android debe ser firmada digitalmente con una clave privada ( keystore ) antes de la distribución. • Permite verificar la integridad y autenticidad de la app.

  • Buenas prácticas: • Mantener el keystore seguro y respaldado. • Evitar usar claves de prueba para apps publicadas en Google Play.

9.2 Versiones y compatibilidad de apps

  • Cada app define en su AndroidManifest y Gradle: • minSdkVersion: versión mínima de Android requerida. • targetSdkVersion: versión para la que la app está optimizada.

  • La app puede ejecutarse en emuladores o dispositivos con API igual o superior a minSdkVersion.

  • Cambios de versión: usar control de versiones en Gradle ( versionCode y versionName ) para actualizaciones en Google Play.

9.3 Depuración y monitoreo

  • Herramientas disponibles en Android Studio: • Depurador: inspección de variables, breakpoints, análisis de logs. • Logcat: seguimiento de mensajes de sistema y errores de la app. • Profiler: análisis de CPU, memoria, red y consumo energético.

  • Permite detectar errores y optimizar el rendimiento antes de publicar.

9.4 Relación entre SDK, IDE, Gradle y Manifest

  • IDE (Android Studio): entorno completo para escribir código, depurar y desplegar apps.

  • SDK: conjunto de herramientas y librerías necesarias para compilar y ejecutar apps.

  • Gradle: automatiza compilación, gestión de dependencias y generación de APK/AAB.

  • AndroidManifest: configura permisos, features y compatibilidad con versiones.

  • La coordinación entre estas capas asegura que la app se compile correctamente, cumpla requisitos del dispositivo y se ejecute sin errores.
