1. Bases de datos objeto-relacionales (ORDBMS)

1.1. Definición

Un ORDBMS es un sistema gestor de bases de datos que combina la estructura relacional con conceptos de la programación orientada a objetos. Permite almacenar tipos de datos complejos, colecciones, herencia, métodos y encapsulación dentro de los tipos de datos, manteniendo compatibilidad con tablas relacionales tradicionales.

1.2. Objetivos principales

  • Integrar modelo relacional y modelo orientado a objetos.

  • Soportar tipos de datos complejos y métodos asociados a los objetos.

  • Permitir la reutilización de estructuras y la herencia de tipos.

  • Facilitar la transición de sistemas relacionales existentes a modelos más ricos y orientados a objetos.

1.3. Tipos de datos objeto en Oracle

  • Definición de tipos:

CREATE [OR REPLACE] TYPE nombre_tipo AS OBJECT (
  atributo1 TIPO,
  atributo2 TIPO,
  ...
  MEMBER FUNCTION/PROCEDURE ...
);

  • Implementación de métodos:

CREATE OR REPLACE TYPE BODY nombre_tipo AS
  <implementación de métodos>
END;

  • Tipos válidos: CHAR, VARCHAR2, NUMBER, DATE, colecciones.

  • Tipos no permitidos: RECORD, LONG RAW.

1.4. Comparación de objetos

  • MAP: define un cálculo sobre los atributos del objeto para comparaciones, devuelve positivo, negativo o cero.

  • ORDER: compara objetos basándose en un atributo escalar (CHAR, NUMBER, DATE, etc.).

  • Permite ordenar o buscar objetos en colecciones y consultas OQL.

1.5. Identificadores de objetos (OID)

  • En Oracle, se asignan automáticamente y son únicos para cada objeto.

  • Garantizan la identificación inequívoca de cada instancia dentro de la tabla de objetos.

1.6. Consultas y manipulación

  • ODL (Object Definition Language): define los tipos de objeto (equivalente a DDL).

  • OQL (Object Query Language): lenguaje declarativo tipo SQL. Permite realizar consultas sobre objetos, incluyendo filtrado, ordenación y operaciones de agregación.

  • Cláusulas principales: INSERT (inserción de objetos), UPDATE (modificación), DELETE (borrado).

1.7. Ventajas de ORDBMS

  • Soporte de tipos de datos complejos y colecciones.

  • Compatibilidad con bases relacionales tradicionales.

  • Permite herencia, métodos y encapsulación dentro de los tipos de datos.

1.8. Desventajas de ORDBMS

  • Incrementa la complejidad del sistema.

  • Mayor coste de mantenimiento.

  • Posible incompatibilidad parcial con sistemas puramente relacionales.

2. Bases de datos orientadas a objetos (OODBMS)

2.1. Definición

Una OODB es un sistema gestor de bases de datos que almacena objetos completos, incluyendo sus atributos y métodos, siguiendo el paradigma orientado a objetos. No depende del modelo relacional y permite representar directamente la lógica y estructura del mundo real en el almacenamiento.

2.2. Objetivos principales

  • Almacenar objetos completos con sus métodos y relaciones.

  • Facilitar el modelado directo del dominio de la aplicación.

  • Permitir operaciones CRUD (crear, leer, actualizar, borrar) sobre objetos de manera directa.

  • Gestionar transacciones y consistencia de datos según el paradigma orientado a objetos.

2.3. Ejemplos

  • NeoDatis: base de datos orientada a objetos escrita en Java, multiplataforma.

  • Db4o: base de datos orientada a objetos para Java y .NET, optimizada para aplicaciones embebidas y de escritorio.

2.4. Operaciones básicas

  • Almacenar objetos:

    • NeoDatis: store(object)

    • Db4o: store(object) seguido de commit()

  • Borrar objetos: delete(object)

  • Modificar objetos:

    • Se recupera el objeto, se modifican sus atributos y se vuelve a almacenar ( store() )

    • Luego se confirma con commit() (Db4o) o la operación se realiza directamente (NeoDatis)

  • Recuperar objetos:

    • Db4o: queryByExample(object) devuelve un ObjectSet.

    • NeoDatis: CriteriaQuery con condiciones Where y métodos gt(), ge(), lt(), le(), equal().

  • Operadores especiales:

    • ALL: recuperar todos los objetos de una clase o tabla.

    • VALUE: extraer atributos individuales de los objetos.

2.5. Consultas y condiciones

  • NeoDatis:

    • CriteriaQuery define la consulta.

    • Clase Where permite aplicar condiciones sobre atributos.

    • Métodos: gt() (mayor que), ge() (mayor o igual), lt() (menor que), le() (menor o igual), equal() (igual).

  • Db4o:

    • queryByExample(object) compara atributos con un patrón de ejemplo.

    • Devuelve un ObjectSet que se puede iterar o filtrar.

2.6. Gestión de transacciones

  • Db4o:

    • Por defecto, se inicia implícitamente al abrir un contenedor.

    • commit(): confirma los cambios de la transacción actual.

    • rollback(): deshace los cambios de la transacción.

  • NeoDatis:

    • Sistema optimista: los cambios se almacenan en memoria hasta confirmación ( commit() ).

    • rollback() deshace los cambios pendientes.

    • En caso de error, se puede recuperar el estado desde el archivo de transacción.

2.7. Ventajas de OODB

  • Representación directa del mundo real mediante objetos completos.

  • Escalabilidad: creación de nuevos tipos de objetos basados en los existentes.

  • Integración natural con lenguajes orientados a objetos.

  • Soporte completo de métodos en objetos.

2.8. Desventajas de OODB

  • Incremento de la complejidad de diseño y mantenimiento.

  • Ausencia de un estándar universal completamente definido.

  • Posible reducción de la seguridad y control de acceso en entornos complejos.

3. Herramientas y clientes de exploración

3.1. Definición

Las bases de datos orientadas a objetos suelen incluir clientes gráficos que permiten navegar, explorar y administrar los objetos almacenados en la base de datos sin necesidad de programar. Estos clientes facilitan la inspección de estructuras, relaciones y colecciones de objetos, así como la ejecución de consultas básicas.

3.2. Objetivos principales

  • Facilitar la visualización de objetos y sus atributos.

  • Permitir la consulta y filtrado de objetos sin escribir código de acceso a datos.

  • Ayudar en depuración y pruebas durante el desarrollo.

  • Administrar la base de datos: inserciones, modificaciones, eliminaciones y exportaciones.

3.3. Ejemplos de clientes

  • NeoDatis ObjectExplorer:

    • Cliente gráfico incluido con NeoDatis.

    • Permite explorar objetos y colecciones en un archivo .odb.

    • Muestra los atributos, relaciones y métodos de los objetos.

    • Permite ejecutar consultas simples y navegar jerárquicamente por los datos.

  • Db4o Explorer: (conceptual, no detallado en las preguntas)

    • Funciona de manera similar, permitiendo inspeccionar ObjectSet y la estructura de objetos en contenedores Db4o.

3.4. Funcionalidades principales

  • Navegación: explorar clases, objetos y colecciones dentro de la base de datos.

  • Visualización de atributos: mostrar el valor de los campos de cada objeto.

  • Filtrado de objetos: aplicar condiciones sobre atributos (por ejemplo, valores mayores que, menores que o iguales).

  • Exportación/importación: mover o copiar datos de una base de datos a otra ubicación.

  • Depuración: verificar que los objetos y relaciones están correctamente almacenados.

3.5. Operadores y métodos relacionados

  • Para mostrar los valores de los atributos de un objeto: operador VALUE.

  • Para filtrar datos en consultas:

    • NeoDatis: CriteriaQuery con Where y métodos gt(), ge(), lt(), le(), equal().

    • Permite consultas completas, parciales o basadas en ejemplos.

  • Para recuperar todos los objetos de una clase: operador ALL.

3.6. Ventajas de estas herramientas

  • Ahorro de tiempo en depuración y pruebas.

  • Visualización directa de la estructura de objetos y relaciones.

  • Reducción de errores al evitar codificación manual de consultas sencillas.

  • Facilitan la migración o copia de bases de datos al inspeccionar y exportar objetos.

3.7. Limitaciones

  • Funcionalidades avanzadas (consultas complejas, agregaciones o joins) pueden requerir código en NeoDatis o Db4o, no solo la interfaz gráfica.

  • No reemplazan el diseño ni la planificación de la base de datos.

  • Dependencia del cliente: algunas versiones pueden no ser compatibles con todas las versiones de la base de datos.

3. Documentación y migración de bases de datos objeto-relacionales y orientadas a objetos

3.1. Importancia de la documentación

  • Documentar una aplicación que accede a una base de datos es crucial para garantizar la portabilidad, mantenibilidad y compatibilidad ante cambios de versión del SGBD o del entorno de ejecución.

  • La documentación adecuada permite a futuros desarrolladores o administradores comprender la configuración, dependencias y métodos de acceso a datos, evitando errores al actualizar o migrar sistemas.

3.2. Elementos que se deben documentar

  1. Versión del sistema gestor de base de datos (SGBD) utilizado

    • Permite garantizar compatibilidad ante actualizaciones o migraciones.

  2. Versión del JRE

    • Documentar la versión de la máquina virtual de Java utilizada, ya que puede afectar la compatibilidad con bibliotecas y el SGBD.

  3. Versión de las bibliotecas de acceso al SGBD

    • Incluye controladores (drivers) y librerías específicas usadas por la aplicación para interactuar con la base de datos.

    • Fundamental para asegurar compatibilidad ante cambios de versión.

  4. Configuraciones específicas del SGBD

    • Opciones de inicialización, parámetros de rendimiento, rutas de archivos de base de datos, etc.

    • Permite replicar el entorno en otro sistema o migrar a un nuevo servidor.

  5. Diagrama de relaciones entre objetos

    • Representación visual de las relaciones entre tablas, vistas y otros objetos utilizados por la aplicación.

3.3. Elementos que NO se deben documentar

  • Configuración temporal del IDE (NetBeans, Eclipse, etc.) que no afecte directamente al acceso a la base de datos.

  • Detalles irrelevantes para la compatibilidad de acceso a los datos o el comportamiento del SGBD.

  • Método de conexión.

3.4. Migración de bases de datos

  • NeoDatis:

    • Para ejecutar la base de datos en otra ubicación, basta con copiar el archivo .odb.

    • También es posible utilizar comandos de export/import, pero la copia directa mantiene la integridad de los objetos y relaciones.

  • Db4o:

    • Similar: se puede trasladar el contenedor de base de datos (.db4o ) a otra ubicación y continuar trabajando.

3.5. Buenas prácticas ante cambios de versión

  • Mantener registro de la versión del SGBD y de las bibliotecas utilizadas.

  • Documentar métodos de acceso y transacciones empleados en la aplicación.

  • Verificar que las consultas y operaciones sobre objetos siguen funcionando tras la actualización.

  • Realizar copias de seguridad antes de migrar la base de datos a otro servidor o versión.

  • Probar la aplicación en un entorno de desarrollo antes de aplicar cambios en producción.

3.6. Objetivos de la documentación y migración

  • Reducir riesgos al actualizar la base de datos.

  • Facilitar la portabilidad y el mantenimiento del sistema.

  • Asegurar la consistencia de los datos y la integridad de los objetos.

  • Evitar errores debidos a dependencias ocultas de versión o configuración del SGBD.
